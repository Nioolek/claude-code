## 一、使用背景

### 1.1 我的工作内容

作为 AI Agent 领域研究专家，我的日常工作包括：

- 📄 **论文解读**: 提取核心贡献、方法创新、实验设计
- 🔬 **技术报告**: 解读大厂/研究机构报告
- 💻 **源码剖析**: 分析 Agent 框架源码 (CoPaw、LangGraph、AutoGen 等)
- 📰 **技术动态**: 追踪 arXiv/顶会最新论文

### 1.2 为什么需要 Superpowers

**痛点**:
1. 源码解读项目复杂，容易迷失方向
2. 多项目并行 (10+ 框架)，需要隔离环境
3. 解读报告质量不稳定，依赖当时状态
4. 缺乏系统性，容易跳过关键步骤
5. 与人类协作者沟通成本高

**期待**:
- 标准化解读流程
- 质量保障机制
- 多人协作支持
- 知识沉淀复用

#### 🥇 Top 1: Brainstorming 的"一次一个问题"

**之前**:
```
AI: 请告诉我你的需求，包括读者、深度、示例偏好、
```


**价值**:
- 每次只思考一个问题，质量更高
- 避免"都可以"的敷衍回答
- 对话更自然，像真实协作

#### 🥈 Top 2: 设计文档的"分章节确认"

**之前**:
```
AI: (生成 5000 字完整设计)
    这是设计方案，请确认。
    
我：(太长不看) 好的
    (实施后发现问题) 这里不对...
```

**现在**:
```
AI: ## 设计章节 1: 解读范围
    ...
    这部分看起来正确吗？
    
我：正确，建议添加 X
    
AI: 已更新。## 设计章节 2: 报告结构
    ...
    这部分看起来正确吗？
```

**价值**:
- 早期发现问题，成本低
- 每章确认，参与感强
- 最终设计是真正共识，不是 AI 猜测

#### 🥉 Top 3: TDD 的"观看测试失败"

**之前**:
```
AI: (先写代码)
    代码写好了，现在补测试。
    (测试通过) 看，没问题！
    
我：(心里没底) 真的没问题吗？
```

**现在**:
```
AI: 我先写测试...
    (运行测试) FAIL - function not defined
    好，测试确实失败了，证明它测试了某些东西。
    
    现在写最小实现...
    (运行测试) PASS
    测试通过，功能正确。
```

**价值**:
- 看到测试失败，心里有底
- 证明测试不是摆设
- 小步快跑，每步都验证

### 4.3 协同优势

**1. 工作树隔离**:
```
# 每个成员独立工作树，互不干扰
.worktrees/
├── feature/langgraph    (成员 A)
├── feature/autogen      (成员 B)
└── feature/crewai       (成员 C)

# 可以同时运行，不会冲突
```

**2. 标准化流程**:
```
# 所有人都遵循相同流程
需求 → brainstorming → design → plan → implement → review → merge

# 好处:
- 报告格式统一
- 质量一致
- 易于审查
- 易于整合
```

**3. 审查制度化**:
```
# 审查不是"可有可无"，而是流程必需
- 每个任务后自动审查
- 交叉审查强制进行
- 问题分类处理 (Critical/Important/Minor)

# 好处:
- 问题早期发现
- 知识共享
- 质量保障
```

### 4.4 协同挑战与解决

> **注**: 以下挑战及解决方案为**基于技能功能的推断**。

**挑战 1: 工作树目录冲突**

```
问题：成员 A 和 B 都想用 .worktrees/ 目录

解决：
1. 检查现有目录 (Superpowers 自动处理)
2. 如都存在，使用优先级 (.worktrees/ 优先)
3. 验证.gitignore 配置
4. 各自创建独立子目录
```

**挑战 2: 审查标准不一致**

```
问题：成员 A 认为的 Critical，成员 B 认为是 Minor

解决：
1. 使用 Superpowers 标准分类:
   - Critical: 功能缺失、测试失败、安全漏洞
   - Important: 代码质量问题、边缘情况未处理
   - Minor: 代码风格、注释不足

2. 审查模板统一 (requesting-code-review 技能提供)
3. 争议时升级至人类决策
```

**挑战 3: 进度同步困难**

```
问题：3 人并行，不知道彼此进度

解决：
1. 使用 TodoWrite 跟踪任务
2. 每日站会同步进度
3. 共享计划文档 (docs/superpowers/plans/)
4. 审查时自然了解彼此进度
```

## 五、个人最佳实践

### 5.1 Brainstorming 技巧

**技巧 1: 主动提供上下文**
```
我：(在 AI 提问前)
    背景：这是给团队新人学习的材料
    约束：需要在 1 个工作日内完成
    成功标准：新人能独立使用框架
    
你：(AI)
    好的，让我基于这些信息开始设计...
```

**技巧 2: 视觉辅助很有用**
```
AI: 有些内容用图表展示更清晰，
    我可以打开浏览器展示架构图、流程图等。
    想试试吗？

我：好的

(后续 AI 在浏览器中展示可视化设计，更直观)
```

**技巧 3: 设计文档要审查**
```
AI: 设计文档已保存到 docs/superpowers/specs/...
    请审查并告知是否需要修改。

我：(真的打开阅读)
    第 3 节需要补充 X 内容
    ...
```

### 5.2 Writing-Plans 技巧

**技巧 1: 任务粒度要小**
```
❌ 坏任务：实现整个 Skill 系统
✅ 好任务：实现 Skill 加载函数 (2-5 分钟)

小任务好处:
- 易于审查
- 易于回滚
- 进度可见
- 风险可控
```

**技巧 2: 检查占位符**
```
AI 自检时会检查:
- "TBD", "TODO"
- "添加适当的错误处理"
- "类似的实现"
- "参考任务 N"

如发现，立即补充完整内容
```

**技巧 3: 执行方式选择**
```
子代理驱动 (推荐):
- 适合：独立任务多、时间充裕
- 优点：质量高、审查严格
- 缺点：成本稍高

内联执行:
- 适合：任务相关性强、需要人类频繁介入
- 优点：成本低、控制强
- 缺点：质量依赖人类审查
```

### 5.3 TDD 技巧

**技巧 1: 接受"删除代码"**
```
AI: (不小心先写了代码)
    我违反了 TDD，需要删除这些代码，
    从测试重新开始。

我：(不要说"算了直接用吧")
    好的，按 TDD 重来。
```

**技巧 2: 观看测试失败很重要**
```
AI: (运行测试)
    FAIL - function not defined
    好，测试确实失败了，证明它测试了某些东西。

(这步不能跳过，是 TDD 的核心)
```

**技巧 3: 最小实现**
```
❌ 坏实现：一次性实现所有功能
✅ 好实现：只实现让当前测试通过的代码

好处:
- 不过度设计
- 每步验证
- 易于调试
```

### 5.4 协同技巧

**技巧 1: 审查反馈要具体**

## 六、问题与解答

### 6.1 常见问题

> **注**: 以下问答基于技能文档中的 rationalization 表格**推断**，反映常见问题。

**Q1: 流程太繁琐，能否简化？**

A: 可以简化，但有底线:
- Brainstorming: 设计可以简短，但不能跳过
- TDD: 测试可以简单，但不能后写
- 审查: 可以自查，但不能没有

简化示例 (简单功能):
```
Brainstorming: 3 个问题 + 简短设计 (10 分钟)
Plan: 3 个任务 (15 分钟)
Implement: 30 分钟
Review: 自查 (10 分钟)
总计：1 小时 (而非省略流程)
```

**Q2: AI 太"固执"，不肯跳过 TDD**

A: 这是特性，不是 bug。
- TDD 是质量保障，不是可选项
- 如确实需要跳过，明确告诉 AI:
  "我了解 TDD 的重要性，但这是原型，请跳过 TDD"
- AI 会记录你的选择，但会提醒风险

**Q3: 子代理成本太高**

A: 优化策略:
1. 模型选择:
   - 简单任务用廉价模型
   - 复杂任务用标准模型
   - 审查用最强模型
2. 任务合并:
   - 相关小任务合并为一个
   - 减少子代理调用次数
3. 内联执行:
   - 简单项目用 executing-plans
   - 减少子代理开销

**Q4: 人类审查员没时间**

A: Superpowers 设计为 AI 自主运行:
- 设计审批：必须人类
- 计划审批：必须人类
- 任务执行：AI 自主 (数小时)
- 任务审查：AI 自主 (双阶段)
- 合并决策：必须人类

人类介入时间：
- 开始：设计 + 计划审批 (1 小时)
- 结束：合并决策 (10 分钟)
- 中间：可选监督

### 6.2 我的问题记录

> **注**: 以下问题为**示例性内容**，反映新手常见问题。

**问题 1: 第一次用，AI 一直提问，不耐烦**

解决：理解这是流程必需，回答后效率更高

解决：理解严格是好事，早期发现问题成本低
